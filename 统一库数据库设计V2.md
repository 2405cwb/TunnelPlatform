# 统一库数据库设计 V2

## 1. 设计结论

当前 V2 设计已经调整为：

```text
以 Word 6.2 成果表为主体 + 增加平台上下文、外键、索引和原始行保留字段
```

这不是简单复刻 Word 原表，也不是继续使用早期临时运行表，而是在 Word 表基础上增强，使其可以支撑一个 PostgreSQL 保存多个工程实例。

## 2. 为什么需要增强 Word 表

Word 原始表本身可以表达站点、环片、病害、图像、点云等成果，但缺少这些平台级上下文：

- 当前数据属于哪个线路。
- 当前数据属于上行还是下行。
- 当前数据属于哪一天采集。
- 当前成果来自哪个站点或区间。
- 当前成果属于哪一次上传批次。
- 覆盖上传时应该删除哪一批旧数据。

因此所有成果表统一补充：

- `RowGuid`：对外展示和 API 使用的稳定 GUID。
- `DatasetId`：上传批次。
- `ProjectInstanceId`：工程实例。
- `StationID`：关联 `STATION.ID`。
- `SourceCategory`：`2D`、`3D`、`gray`、`image`、`point-cloud`。
- `SourceTable`：来源 SQLite 表名或来源目录。
- `CreatedAt`：入库时间。
- `RawJson`：原始数据行 JSON，保留暂未映射字段。

## 3. 平台上下文表

### projects

表示线路或逻辑工程，例如 `北京地铁6号线`。

关键字段：

- `Id`
- `ProjectName`
- `CreatedAt`
- `UpdatedAt`

### project_instances

表示一次具体工程实例：

```text
线路名称 + 上下行 + 采集日期
```

关键字段：

- `Id`
- `ProjectId`
- `Direction`
- `CollectionDate`
- `DisplayName`
- `CreatedAt`
- `UpdatedAt`

唯一约束：

```text
ProjectId + Direction + CollectionDate
```

### STATION

承接 Word `STATION`，但语义升级为“工程实例下的站点/区间快照”。

关键字段：

- `ID`
- `StationGuid`
- `ProjectId`
- `ProjectInstanceId`
- `EntityCode`
- `LineName`
- `Direction`
- `CollectionDate`
- `BegStation`
- `EndStation`
- `BegMileage`
- `EndMileage`
- `StationType`
- `TunnelType`
- `StationNum`

### entity_datasets

表示某个工程实例下某个站点/区间的一次最终导入结果。

关键字段：

- `Id`
- `ProjectInstanceId`
- `StationID`
- `StationGuid`
- `StorageRelativePath`
- `Has2D`
- `Has3D`
- `GrayImageCount`
- `DiseaseImageCount`
- `PointCloudFileCount`
- `DiseaseCount`
- `ImportedAt`

覆盖上传时以 `entity_datasets` 为根，级联删除旧成果。

## 4. Word 增强成果表

| 表 | 数据来源 | 当前用途 |
| --- | --- | --- |
| `RING_LOC` | 2D `RING_TUNNEL`；无 2D 时 3D `CIRCLED_FLAKE` | 环片位置和里程叠加 |
| `RING_FIT` | 3D `FIT_DIAMETER` | 收敛、长短轴、拟合指标 |
| `HORSE_DIST` | 3D `HORSE_DIST` | 水平距离 |
| `INTER_RING_PLATFORM` | 3D `PLAT_FORM` | 环间错台 |
| `INNER_RING_PLATFORM` | 3D `CIRCLED_FLAKE` | 环内错台 |
| `LIMIT_CHK_DATA` | 3D `LIMIT_DATA` | 限界检测 |
| `DISEASE_CHK_DATA` | 2D `BASIC_DISEASE`；无 2D 时 3D 病害表 | 病害列表、统计、高清图联动 |
| `SEC_PTCLOUD_TATA` | `04点云` 文件目录 | 点云文件索引，保留 Word 原始拼写 |
| `SEC_PTCLOUD_DATA` | 视图 | 对 `SEC_PTCLOUD_TATA` 做兼容别名 |
| `IMAGE_DATA` | `03灰度图`、`05二维病害高清图`、2D `COMBINE_IMAGE` | 图片连续浏览、病害高清图 |

## 5. 二维优先规则

二维优先是当前导入逻辑的重要原则：

- 存在 `01二维数据` 时，`DISEASE_CHK_DATA` 使用 2D `BASIC_DISEASE`。
- 存在 `01二维数据` 时，`RING_LOC` 使用 2D `RING_TUNNEL`。
- 没有 `01二维数据` 时，才使用 3D 病害表和 3D `CIRCLED_FLAKE` 补充。
- 3D 的结构表仍正常写入，因为这些表没有 2D 等价来源。

## 6. 暂未完全映射的数据

当前未强行落入 Word 成果表的来源包括：

- 2D `SECTION_TABLE`
- 2D `PLATFORM_TABLE`
- 2D `TUNNEL_INFO`
- 2D `TUNNEL_LOCATION`
- 2D `MILEAGE_CONVERT`
- 2D `IMAGEDISEASE_INFO`
- 3D `SECTION`
- 3D `CATENARY`
- 3D `CLEARANCE`
- 3D `RECT_DIST`
- 3D `HORSE_FIT`
- 3D `STATION_DIST`

处理原则：

- 不为了“看起来完整”随意塞字段。
- 已映射成果保留 `RawJson`，方便回溯原始行。
- 后续确认业务含义后，再决定是扩展现有 Word 表，还是新增 Word 基础上的补充表。

## 7. 索引设计

### STATION

```sql
CREATE UNIQUE INDEX IF NOT EXISTS "UX_STATION_Instance_Entity"
ON "STATION" ("ProjectInstanceId", "EntityCode");

CREATE INDEX IF NOT EXISTS "IX_STATION_Instance_Type_Mileage"
ON "STATION" ("ProjectInstanceId", "StationType", "BegMileage", "EndMileage");
```

### DISEASE_CHK_DATA

```sql
CREATE INDEX IF NOT EXISTS "IX_DISEASE_Instance_Station_Mileage"
ON "DISEASE_CHK_DATA" ("ProjectInstanceId", "StationID", "DiseaseMileage");

CREATE INDEX IF NOT EXISTS "IX_DISEASE_Instance_Name_Mileage"
ON "DISEASE_CHK_DATA" ("ProjectInstanceId", "DiseaseName", "DiseaseMileage");
```

### IMAGE_DATA

```sql
CREATE INDEX IF NOT EXISTS "IX_IMAGE_Instance_Station_Type_Mileage"
ON "IMAGE_DATA" ("ProjectInstanceId", "StationID", "ImageType", "BegMileage", "EndMileage");

CREATE INDEX IF NOT EXISTS "IX_IMAGE_Instance_Station_Disease_Mileage"
ON "IMAGE_DATA" ("ProjectInstanceId", "StationID", "DiseaseType", "CenterMileage");
```

### 结构成果表

每张结构成果表都围绕：

```text
ProjectInstanceId + StationID + Mileage/BegMileage
```

建立查询索引。

## 8. 展示平台查询支撑

当前数据库可以支撑：

- 工程实例列表。
- 工程概览。
- 工程里程范围。
- 当前工程病害统计。
- 当前区间病害统计。
- 当前区间全部病害列表。
- 按病害类型和里程范围筛选。
- 二维灰度图连续浏览。
- 环片叠加。
- 双击病害查看高清图。
- 点云文件按帧号浏览的后续扩展。

## 9. 当前结论

V2 的最终口径是：

```text
数据库主体遵循 Word 设计表，平台能力通过上下文字段、外键和索引增强实现。
```

这条路线既能满足工作要求，又比旧临时表方案更容易给同事解释和交付。
